## 🔴 最嚴重：`episode_checkout_click` / `episode_play_click`

### 規格（7/7 更新）
> 影片列表的 episode 卡片樣式——按照 PRD，不可播放時是不可點擊狀態，不會有事件。**這個位置的點擊事件只會有 `episode_play_click`**（`position_click_type = episode_card`）。
> 介紹頁上的按鈕——因為都可以點擊，所以會觸發兩種事件：`episode_checkout_click` & `episode_play_click`（`position_click_type = play_button`）。

白話：**只有播放按鈕（CTA）能觸發 `episode_checkout_click`；單集卡片（episode_card）永遠只能是 `episode_play_click`。**

### iOS：❌ 違反規則，且已找到具體觸發條件

`LearningsPackViewModel.swift` 的 `logEpisodeCTAClick` 判斷送哪個事件，依據的是**整個系列/package 的** `content.accessStatus`，跟這次點擊是來自播放按鈕（`.playButton`）還是單集卡片（`.episodeCard`）**完全無關**：

```swift
switch content.accessStatus {
case .paymentRequired:
    logService.clickEpisodeCheckout(..., clickType: clickType, ...)   // 不管 clickType 是什麼都送 checkout
case .pendingActivation, .active:
    logService.clickEpisodePlay(..., clickType: clickType, ...)
case .accessUnavailable, .unknown:
    break
}
```

而單集卡片是否可點擊，用的是**該集自己的** `episode.playable` 欄位（`VideoTableCell.swift`：`if isPlayable { onTapVideo?(video.id) }`）——這是一個**獨立於系列層級 `accessStatus` 的欄位**。也就是說，只要後端讓某一集是免費試看（`playable:true`），即使整個系列 `accessStatus == .paymentRequired`，這一集的卡片依然點得下去，點下去後 `logEpisodeCTAClick` 還是照系列層級狀態走 `.paymentRequired` 分支，送出：

```json
// 實際會送出（違反規格）：
episode_checkout_click {
  "position_page": "學習館影音介紹頁",
  "position_title": "溫美玉x魏瑛娟魔法作文趴",
  "position_type": "learnings/series",
  "position_id": "24",
  "position_click_type": "episode_card",   // ← 規格明講這裡永遠只能是 episode_play_click
  "id": "100",
  "name": "1.免費預覽"
}
```

**修法方向**（僅供參考，未實作）：`logEpisodeCTAClick` 應該在 `clickType == .episodeCard` 時無條件走 `episode_play_click`，`accessStatus` 分支只保留給 `.playButton` 使用。

### Android：✅ 符合規則，但有一個獨立的小 bug

`playEpisode` 的分支判斷順序是「先看 `fromActionButton`，才看 `isUnpaid`」：

```kotlin
fromActionButton && state.uiState.isUnpaid -> {
    logEpisodeCheckoutClick(episodeId, state); handleUnpaid()
}
else -> {
    logEpisodePlayClick(episodeId, fromActionButton, state)
    executePlayEpisode(episodeId, allowActive)
}
```

單集卡片點擊固定傳 `fromActionButton = false`（`LearningsPackOverviewFragment.kt` 的 `onEpisodeClick`），且卡片本身在 `playable=false` 時就沒有 `clickable` modifier——**兩層防護都到位**，`episode_checkout_click` 確實只可能從播放按鈕觸發，完全符合 7/7 規則。

但抓到一個新的小 bug：`confirmActivation()`（使用者在「需要開通」對話框按確認後的重試路徑）：

```kotlin
fun confirmActivation() {
    ...
    playEpisode(dialog.episodeId, allowActive = true)   // 沒有帶 fromActionButton=true，預設變成 false
}
```

如果原始點擊其實是**播放按鈕**、遇到需要開通的彈窗、使用者按下確認後重新播放，這次重試會被誤判成 `fromActionButton=false`，導致 `episode_play_click` 的 `position_click_type` 被誤標成 `"episode_card"`，儘管整個流程從頭到尾都是點播放按鈕。跟上面的規則違反無關，但也是可以順手修的參數錯誤來源。

## 🔴 Zone 1-5 白名單問題（跨事件共通）

### 規格依據
6/16 決策：「新的這幾個 UI type 點擊事件，都用『新的參數結構』，會與原先的 UI type 點擊事件不同」；`click_album`「不更改」；每個 Zone 的 `target_type` 值域說明只列 `learnings/series`／`learnings/packages`／`learnings/editorials`／`webview`／`browser`，從未出現 `album`。

### iOS：✅ 有 gating

`FirebaseLog.LearningsHomeClickTarget.init?(cardType:target:title:)` 對不認得的 `cardType`（例如 `album`）回傳 `nil`，呼叫端用 `if let target = ... { 送事件 }` 包住——`album` 類型完全不會觸發這 4 個新事件，只會走舊的 `click_album`。

**新發現的副作用**：如果 banner/卡片的 `type` 既不是白名單裡的 learnings 類型、也不是 `album`（例如 `curation`／`newestAlbum`／`questCampaign` 這類其他合法 `DispatchType`），那麼**新舊事件都不會發送**，這次點擊完全沒有任何 GA 紀錄。這不算違反規格（規格沒定義這個情況），但值得跟 PM 提一下這是個追蹤覆蓋率的缺口。

### Android：❌ 無 gating

`CustomLayoutAdapter.kt` 裡 4 個 Zone 的 `onSel*Click`／`onSeeAllClick` 都是**無條件**呼叫 `cardElementClick(...)`，送完新事件後才呼叫 `onRedirectWithTypeAndTarget`（裡面才有針對 `album` 額外送 `click_album` 的邏輯）。這代表：**只要學習館首頁的區塊真的出現 `type=album` 的項目，Android 會同時送出兩個事件**（新事件 `target_type=album` ＋ 舊事件 `click_album`），與 6/16 決策的「新舊事件互斥、新事件只對應學習館目的地」明顯不符。由於 `CustomLayoutAdapter` 是全站共用元件（`HomeFragment`／`FreemiumFragment`／`GalleryEventFragment` 等都在用），這個問題不只可能出現在學習館首頁。

## 🟠 `bulletCardFlow_click` 的 header 箭頭——Android 送成完全不同的事件

### 規格推論
文件對 Zone 3／5 都只寫了卡片點擊情境，沒有另外寫箭頭情境；但 Zone 2／4 都明確寫「點擊【>】進入內頁」沿用同一個事件名。合理推論：箭頭沿用卡片同一事件名，是全部 5 個 Zone 共通的設計，文件只是没重複寫。

### iOS：✅ 箭頭與卡片共用同一事件名

```swift
SectionHeaderView(title: listObj.title, showArrow: ...) {
    if let target = FirebaseLog.LearningsHomeClickTarget(arrowType: ..., target: ...) {
        LogService.shared.clickLearningsHomeSection(section: .bulletCardFlow, ...)  // → bulletCardFlow_click
    }
}
```

### Android：❌ 箭頭走的是舊的通用事件

```kotlin
// BulletCardFlowViewHolder
onActionClick = {
    onSectionIconClick(type = viewInfo.type, target = viewInfo.target, sectionTitle = viewInfo.title)
}
// → logService.sectionIconClick(positionPage, positionType) → 送 "section_icon_click"
```

**實際送出結果對照：**

| | iOS 箭頭點擊 | Android 箭頭點擊 |
|---|---|---|
| 事件名稱 | `bulletCardFlow_click` | `section_icon_click` |
| 參數 | `position_page`、`position_title`、`target_id`、`target_type`、`target_title` | 只有 `position_page`、`position_type`（值是 section 標題，語意上跟其他事件的 `position_type` 完全不同） |

Android 這條路徑跟其他一堆舊版 UI type（`ImageCardList`／`LabelCardList`／`CircleCardList` 等）共用同一個 `onSectionIconClick`，Medium／Large 兩個 Zone 已經在 6/16 決策時改接到新的 `onSeeAllClick`，**BulletCardFlow 看起來是當時漏改的**。

## 🟡 Zone 2（我的學習庫）header 箭頭的 `target_id`

### 規格
「我的學習庫，**無 target id，則不帶**」——這個 key 應該完全不存在。

### iOS：✅ 完全不存在

```swift
case .myLibraryArrow:
    parameters["target_type"] = "learnings/my-library"
    parameters["target_title"] = "全部"
    // 沒有設 target_id，這個 key 完全不會出現在 dictionary 裡
```

### Android：⚠️ Key 存在、值是 null

```kotlin
fun cardElementClick(..., targetId: String?, ...) {
    firebaseLog.logEvent(eventName, data = bundleOf(
        "target_id" to targetId,   // targetId 是 null，但 bundleOf 還是會建立這個 key
        ...
    ))
}
```

`bundleOf(...)` 對 null 值一樣會 `putString(key, null)`，key 會存在（`bundle.containsKey("target_id") == true`），只是值是 null——跟「完全不帶」在 Bundle 層級是不一樣的兩件事。同一支檔案裡 `position_title` 跟 `learningsPackClick` 的 `position_id` 都有正確用 `.also{}`／`?.let{}` 做「非 null 才加 key」的保護，只有這個 `cardElementClick` 的 `target_id` 沒有做。實際上 Firebase SDK 送到後端前可能會把 null 參數濾掉，但至少在 App 自己的除錯工具（讀 `Bundle.keySet()`）跟本地紀錄裡，這個 key 是看得到的。

## 其餘完全符合規格的部分

- **`heroRectangleCarousel_click`**：兩邊事件名稱、參數、資料來源都對，`position_title` 正確地完全不存在（Zone 1 沒有標題列）。
- **`mediumRectangleLabelledCardList_click`（Zone 4 策展清單）／`largeRectangleLabelledCardList_click`（Zone 3）**：卡片與箭頭情境，兩邊都完全符合規格，包含 `position_title` 用區塊自己的標題（不是被點項目標題）這個容易寫錯的細節。
- **`learnings_series_click`／`learnings_package_click`**：我的學習庫情境（`position_title` 固定字串「我的學習庫」、無 `position_id`）跟策展列表頁情境（`position_title` 動態帶策展標題、`position_id` 存在），兩邊程式碼都完全對得上規格範例值。

## 建議修復順序

1. **iOS：`episode_checkout_click`／`episode_play_click`**——`logEpisodeCTAClick` 判斷邏輯要以 `clickType` 優先於 `accessStatus`；`episode_card` 永遠強制走 `episode_play_click`。這條有實際的 GA 資料污染（未付費系列的免費試看集會被誤記成 checkout）。
2. **Android：`bulletCardFlow_click` header 箭頭**——改接到跟 Medium/Large 一樣的 `onSeeAllClick`，事件名跟參數才會對齊。
3. **Android：Zone 1-5 的 `album` 白名單**——`cardElementClick` 呼叫前應該先檢查 `type` 是否為學習館相關目的地，避免跟 `click_album` 雙重觸發。
4. **Android：Zone 2 header 箭頭 `target_id`**——`cardElementClick` 的 bundle 建構比照 `position_title` 的寫法，改成非 null 才加入 key。
5. **Android：`confirmActivation()` 續期重試**——補上 `fromActionButton = true`，避免 `position_click_type` 被誤標成 `episode_card`。

# ADR-001-選擇技術棧

**狀態：** accepted

## 背景 (Context)

[[EPIC-001-待辦事項管理]] 是一個單一使用者、資料量小（預期 < 1000 筆/人）的內部工具 MVP，目標是盡快上線驗證使用行為，且部署環境需簡單（可能是內部一台機器或單一容器），不應該引入需要額外基礎設施（如獨立資料庫伺服器）的方案。

## 考慮過的選項 (Options Considered)

### 選項 A：Node.js + Express + PostgreSQL
- 優點：生態成熟、之後擴充多使用者/協作功能容易
- 缺點：MVP 階段需要額外維運一個 PostgreSQL 服務，違背「盡快上線」與「部署簡單」的限制

### 選項 B：Node.js 內建 http 模組 + 檔案內記憶體儲存（無外部相依套件）
- 優點：零安裝相依、`npm install` 都不需要即可執行，最符合 MVP 求快的限制；程式碼行數少、易讀
- 缺點：重啟服務會遺失資料；未來要換成真正的資料庫需要重構儲存層

### 選項 C：Python + Flask + SQLite
- 優點：SQLite 免維運
- 缺點：與團隊現有前端工具鏈（Node/npm）不一致，徒增技術棧種類

## 決策 (Decision)

採用**選項 B**：Node.js 內建 `http` 模組實作 REST API，資料先以記憶體陣列儲存（模組化的 store 層），欄位設計直接對齊 [[Database Schema]]，未來要換成 SQLite/PostgreSQL 時只需替換 store 層實作，API 與前端不受影響。

## 理由 (Rationale)

- MVP 目標是驗證使用行為，不是打造長期架構；零相依套件可讓任何人 `node src/backend/server.js` 就跑起來，符合「部署簡單」的限制。
- Store 層（`src/backend/db.js`）以函式介面封裝（`list/create/update/remove`），呼叫端不知道底層是記憶體還是資料庫，滿足「未來可替換」的需求，不需要現在就決定資料庫種類。

## 後果 (Consequences)

- 正面：新人 clone 專案後不需要安裝資料庫、不需要 `npm install` 即可跑起完整功能，降低 onboarding 成本。
- 負面 / 需要承擔的技術債：服務重啟即遺失資料，僅適用於 Demo/開發環境；若要正式上線多使用者版本，必須先完成「換成持久化儲存」的技術債（在 [[Roadmap]] 2026 Q4 前必須處理）。

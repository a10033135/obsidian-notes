# 共通規範 (SDD Conventions)

本文件定義 Spec-Driven Development (SDD) 在本專案中的運作規則。所有角色（PM / UIUX / 工程）在 `docs/` 下新增或修改文件前，請先讀過這份文件。

## 1. SDD 是什麼、為什麼要用

Spec-Driven Development：**先寫規格、再寫程式**。任何會影響超過一人、或會被日後回頭查證的決策，都要先落成 `docs/` 底下的文件，程式碼與 PR 只是規格的實作結果。

好處：
- 決策有歷史紀錄可查（why），不必靠口頭傳遞或翻 Git log 猜測。
- PM / UIUX / 工程三方在同一份 Obsidian Vault 裡用雙向連結互相參照，不必在三個不同工具（Confluence / Figma / Notion）之間對照。
- 新人 onboarding 只需要照 `01 → 02 → 03 → 04` 的順序讀一次。

## 2. 資料夾與角色對應

| 資料夾 | 主責角色 | 內容 |
|---|---|---|
| `00-Meta` | 全體 | 共通規範、Tags、Templates |
| `01-Product` | PM | PRD、Epic、Roadmap、User Stories |
| `02-Design` | UIUX | Design System、User Flow、Figma 連結 |
| `03-Engineering` | 工程 | Architecture (ADR)、API Spec、Database Schema |
| `04-Features` | 跨角色協作 | 單一功能的規格書，彙整前三者的結論 |

## 3. 文件命名規則

| 類型 | 格式 | 範例 |
|---|---|---|
| PRD | `PRD-<編號>-<產品或主題>.md` | `PRD-001-TaskFlow待辦事項管理.md` |
| Epic | `EPIC-<編號>-<名稱>.md` | `EPIC-001-待辦事項管理.md` |
| User Story | `US-<編號>-<動作>.md` | `US-001-建立待辦事項.md` |
| ADR | `ADR-<編號>-<決策標題>.md` | `ADR-001-選擇技術棧.md` |
| Feature Spec | `FEAT-<編號>-<功能名稱>.md` | `FEAT-001-待辦事項CRUD.md` |

編號在同類型文件中遞增，不重複使用、不因文件廢棄而回收。

## 4. 狀態標籤 (Status)

每份規格類文件的 frontmatter 都要有 `status` 欄位，值只能是：

- `draft` — 草稿，內容可能大幅變動
- `in-review` — 已提交跨角色審閱
- `approved` — 已核准，可作為開發依據
- `in-progress` — 對應功能開發中
- `shipped` — 已上線
- `deprecated` — 已棄用，保留供歷史查詢

## 5. SDD 工作流程

```
1. PM 在 01-Product 寫 PRD（status: draft）
2. PM 拆出 Epic 與 User Stories，連結回 PRD
3. UIUX 在 02-Design 依 User Stories 產出 User Flow 與畫面規格，連結回 Epic
4. 工程在 03-Engineering 寫 ADR 決定技術方案、定義 API Spec 與 Database Schema
5. 三方共同在 04-Features 彙整成一份 Feature Spec（status 轉為 approved）
6. 工程依 Feature Spec 實作 src/ 程式碼，PR 描述連結回 FEAT 文件
7. 上線後，PRD / Epic / Feature Spec 的 status 更新為 shipped
```

## 6. 雙向連結原則

- 下游文件一律連回上游文件（Feature Spec 連 PRD/ADR/Design，User Story 連 Epic），不要反過來要求上游連結所有下游，避免上游文件因為連結太多而難以維護。
- 使用 Obsidian `[[wikilink]]`，不要用相對路徑的 Markdown 連結，才能在檔案搬移時自動更新（見 `.obsidian/app.json` 的 `alwaysUpdateLinks`）。

## 7. 新增文件的步驟

1. 到 `docs/00-Meta/Templates/` 複製對應樣板。
2. 依第 3 節命名規則命名，放進對應角色資料夾。
3. 填寫 frontmatter（tags、status、owner、created）。
4. 補上與上游文件的 `[[wikilink]]`。
